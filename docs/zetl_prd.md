# 基于Zig 0.17的高性能多源数据归集ETL系统 PRD V1.0
**项目代号**：zETL
**适用场景**：数十套阿里云PolarDB MySQL小程序商城 → 统一数据归集，支撑代理商订单/佣金查询
**核心定位**：用Zig 0.17原生开发的轻量、高吞吐、无GC、低资源占用的CDC实时ETL引擎，替代商用ETL工具，完全自主可控，适配多库归集业务。

---

## 一、文档概述
### 1.1 文档目的
定义基于Zig 0.17自研ETL系统的完整产品需求、技术边界、功能规格与验收标准，指导开发、测试、运维落地，满足多商城PolarDB数据实时归集与佣金计算的业务诉求。

### 1.2 业务背景与痛点
1. 现有商用ETL工具存在授权成本、定制化能力受限、Java系引擎资源占用高、GC卡顿影响大促同步稳定性等问题；
2. 几十套独立PolarDB商城数据孤岛，代理商无法实时查看分渠道订单与佣金；
3. 传统定时轮询同步对源库压力大，易漏单、重复计算佣金；
4. 需一套**极低资源占用、无运行时依赖、单文件部署、高并发稳定**的同步引擎，适配低成本ECS长期运行。

### 1.3 项目目标
1. 功能对齐ETLCloud核心能力：多源MySQL/PolarDB CDC实时同步、可视化配置、ETL转换、幂等写入、自动对账、监控告警；
2. 性能超越Java系ETL：4核8G单机承载≥100条同步链路，单链路吞吐≥5000 TPS，端到端延迟≤10s，无GC停顿；
3. 部署极简：单二进制文件启动，无JDK/运行时依赖，内存占用控制在2GB以内；
4. 业务适配：原生支持多商城打标、代理商佣金实时计算、分库数据对账、异常数据分流。

### 1.4 范围界定
#### 本期Must实现
- 数据源：阿里云PolarDB MySQL / 原生MySQL 5.7/8.0 作为源端与目标端
- 同步模式：全量初始化 + CDC binlog实时增量同步
- 转换能力：字段映射、常量注入、数据清洗、表达式计算、数据过滤
- 业务能力：自动追加mall_id、订单佣金计算、幂等去重、脏数据分流
- 运维能力：Web可视化控制台、同步监控、延迟告警、定时自动对账
- 部署形态：单机部署，单二进制分发

#### 二期Not Now
- 分布式集群高可用
- Kafka/RocketMQ消息队列中间件对接
- 非MySQL数据源（PostgreSQL、MongoDB、API）
- 多租户分级权限、代理商直接登录查看数据

### 1.5 术语定义
| 术语 | 说明 |
|------|------|
| CDC | Change Data Capture，基于binlog的增量数据变更捕获 |
| mall_id | 每套小程序商城唯一编码，多源数据隔离标识 |
| GTID | MySQL全局事务ID，用于binlog断点续传位点记录 |
| Sink | 数据写入目标端组件 |
| 全量初始化 | 首次上线同步历史存量数据 |
| 对账任务 | 离线校验源库与归集库数据一致性的定时任务 |

## 二、用户角色与核心业务流程
### 2.1 用户角色
1. **数据管理员**：新增/管理数据源、配置同步任务、设置佣金规则、查看全量监控、处理异常数据
2. **运维工程师**：部署升级服务、查看系统运行状态、处理告警、排查同步故障
3. **业务代理商**：不登录本系统，通过总后台业务系统读取归集库数据

### 2.2 核心业务流程
1. **新增商城接入**：管理员在Web控制台录入PolarDB连接信息 → 系统校验连通性与binlog权限 → 创建同步任务 → 执行全量历史数据初始化 → 自动切换为CDC实时增量同步 → 数据自动追加mall_id写入归集库
2. **日常实时同步**：商城订单生成/更新 → PolarDB生成binlog → zETL引擎解析捕获 → 执行ETL转换与佣金计算 → 幂等写入总归集库
3. **数据对账兜底**：每日凌晨定时任务 → 按mall_id统计源库与目标库订单数/金额 → 差值超阈值生成异常报表并告警
4. **故障告警处理**：同步延迟超标、链路断开、写入失败 → 系统自动重试 → 重试失败推送企业微信告警 → 运维介入处理

## 三、整体技术架构（Zig 0.17 技术栈约束）
### 3.1 选型依据：Zig 0.17 做ETL的核心优势
1. **无GC、无运行时抖动**：手动内存管理+内存池复用，大促高峰无GC停顿，同步延迟稳定；
2. **极低资源占用**：单二进制文件，启动内存<50MB，4核8G可承载上百条同步链路，服务器成本远低于Java系Flink/DataX；
3. **原生高性能异步IO**：Zig 0.17 异步并发模型，单进程并行处理多路数据库连接，吞吐远超同资源Go/Java实现；
4. **部署极简**：无JDK/虚拟机依赖，单可执行文件直接运行，容器化/裸机均可；
5. **内存安全**：编译期内存检查，避免野指针、内存泄漏，7×24小时长期稳定运行。

### 3.2 系统分层架构
```
┌─────────────────────────────────┐
│  Web管理控制台（Zig后端 + 前端） │  任务配置、监控、告警、对账
├─────────────────────────────────┤
│      调度与元数据管理层         │  任务调度、位点管理、元数据存储
├─────────────────────────────────┤
│      ETL转换引擎层              │  字段映射、清洗、计算、过滤
├─────────────────────────────────┤
│  CDC采集层  │  Sink写入层       │  MySQL binlog解析 / 目标端写入
├─────────────────────────────────┤
│      基础组件层                 │  连接池、内存池、日志、告警
└─────────────────────────────────┘
```

### 3.3 核心技术栈
- **核心引擎**：Zig 0.17 原生开发，异步IO，无第三方重型依赖
- **binlog解析**：基于MySQL binlog协议原生实现，或绑定成熟C库做Zig封装
- **Web控制台**：Zig HTTP服务 + 轻量前端（HTMX + Alpine.js，无构建流程）
- **元数据存储**：单文件SQLite，无需额外数据库
- **部署形态**：单二进制可执行文件，内置所有资源

### 3.4 部署拓扑（适配阿里云PolarDB场景）
- 单台阿里云ECS部署zETL服务，与所有PolarDB实例同地域同VPC内网互通
- 源端：数十套PolarDB MySQL（商城业务库）
- 目标端：独立RDS/PolarDB（总归集库）
- 告警通道：企业微信机器人Webhook

## 四、核心功能需求
### 模块1：数据源管理
1. **数据源类型**：原生支持MySQL / 阿里云PolarDB MySQL，可视化填写连接参数（地址、端口、库名、账号、密码、mall_id）
2. **连通性校验**：一键测试网络连通、账号权限、binlog配置（`binlog_format=ROW`、`binlog_row_image=FULL`）
3. **批量导入**：支持Excel批量导入数十套商城数据源，自动分配mall_id
4. **数据源生命周期**：编辑、停用、删除（仅未绑定任务的数据源可删除）
5. **安全存储**：数据库密码AES加密存储，页面不明文展示

### 模块2：CDC实时同步引擎（核心模块）
1. **双模式同步**
   - 全量初始化：首次启动自动SELECT分批拉取历史数据，不锁表、不影响源库业务
   - 增量CDC：全量完成后自动切换为binlog实时监听，无缝衔接无数据断层
2. **断点续传**
   - 基于GTID持久化同步位点，存储于本地SQLite
   - 服务重启、网络中断恢复后，从上次位点继续同步，不丢数据、不重复
3. **多任务并行**
   - 单进程内支持数十至上百条同步链路并行运行，独立隔离，单任务故障不影响全局
   - 支持动态新增/停止任务，无需重启服务
4. **事件处理**
   - 完整支持INSERT/UPDATE/DELETE三类binlog事件
   - 删除事件默认软删除，写入目标端`is_delete=1`，不物理删除数据
5. **流量控制**
   - 全量阶段可配置限速，避免打满源库带宽
   - 支持大促期间临时降级限流配置

### 模块3：ETL数据转换引擎
1. **常量字段注入**：每条数据自动追加`mall_id`、`sync_time`同步时间戳、`source_type`数据源类型
2. **字段映射与标准化**
   - 可视化配置源字段与目标字段映射关系
   - 支持字段重命名、类型转换、空值填充、默认值设置
3. **数据清洗**
   - 手机号/身份证脱敏、日期格式统一、字符串去空格
   - 脏数据识别：字段缺失、格式错误自动标记
4. **业务计算能力**
   - 表达式引擎：支持加减乘除、条件判断、字符串处理
   - 佣金计算：可视化配置分佣规则（固定比例、阶梯分成），实时计算`agent_commission`
   - 支持关联维度表（代理商关系表、商品分佣表）做字段补全
5. **数据过滤**：支持按条件过滤作废订单、测试订单、指定状态数据
6. **脏数据分流**：转换失败、规则不匹配的数据写入脏数据表，记录错误原因，支持人工复核重入

### 模块4：数据写入Sink模块
1. **目标端支持**：MySQL / PolarDB MySQL，支持分库分表写入
2. **幂等写入**
   - 基于`mall_id + 业务主键`（如order_no）做联合唯一索引
   - 冲突策略可选：按更新时间覆盖、忽略、报错
3. **批量写入优化**
   - 攒批提交，可配置批量大小、刷盘间隔
   - 自动事务控制，异常回滚，保证数据一致性
4. **写入限流**：可配置目标端最大写入QPS，避免打垮归集库

### 模块5：调度与数据对账模块
1. **定时调度**：内置Cron表达式调度器，支持定时全量对账、定时全量同步
2. **自动对账能力**
   - 每日凌晨按mall_id维度统计源库订单数、订单总金额、用户数
   - 与目标归集库统计值对比，计算差值
   - 差值超过可配置阈值（订单数差>5、金额差>100元）标记异常
3. **对账报表**：生成每日对账明细，支持导出CSV，留存90天

### 模块6：监控告警模块
1. **全局监控大盘**
   - 总览：运行中任务数、今日同步总条数、平均延迟、异常任务数
   - 单任务详情：对应mall_id、当前同步位点、延迟秒数、今日成功/失败条数
2. **日志系统**
   - 运行日志、错误日志、binlog解析日志分级存储
   - 支持按任务、时间、级别检索
3. **告警规则**
   - 同步延迟超阈值（默认30s预警、60s告警）
   - 任务异常停止、连接断开
   - 对账数据差值超标
   - 脏数据量持续增长
4. **告警渠道**：企业微信机器人Webhook推送，支持自定义告警模板

### 模块7：Web管理控制台
1. 单页轻量控制台，无需前端构建，随二进制内置
2. 功能菜单：数据源管理、同步任务管理、转换规则配置、监控大盘、对账报表、系统设置
3. 账号权限：管理员账号+只读运维账号，密码加密存储
4. 操作审计：所有配置修改、任务启停操作留痕，可追溯

### 5.1 性能指标（Zig核心优势体现）
| 指标项 | 要求值 | 说明 |
|--------|--------|------|
| 单机最大同步任务数 | ≥100条 | 4核8G EC2配置下 |
| 单链路同步吞吐 | ≥5000行/秒 | 单表CDC增量同步 |

| 百条任务运行内存 | ≤2GB | 远低于Java/Go同规模实现 |
| CPU使用率 | 4核下≤60% | 满负载百条任务运行 |
| 全量初始化速度 | ≥10万行/秒 | 单表批量同步 |

### 5.2 可靠性与数据一致性
1. 数据不丢不重：异常恢复后无数据丢失、无重复写入
2. 故障自动重试：网络抖动、数据库闪断自动重试，重试间隔指数退避
3. 单任务隔离：单条同步链路崩溃不影响其他任务，整体服务可用性≥99.9%
4. 位点持久化：每1s刷盘一次同步位点，断电恢复最大丢失数据量≤1s

### 5.3 兼容性
1. 源端兼容：MySQL 5.7、8.0；阿里云PolarDB MySQL 5.7/8.0
2. 部署兼容：Linux x86_64/ARM64、Docker容器、OpenEuler等信创系统
3. binlog兼容：支持ROW格式，支持FULL/MINIMAL镜像

### 5.4 安全性
1. 数据库连接凭证加密存储，禁止明文落盘
2. Web控制台支持HTTPS、会话超时、密码强度校验
3. 操作审计日志不可删除，留存90天
4. 源库同步账号仅授予最小权限：`SELECT、REPLICATION SLAVE、REPLICATION CLIENT`

### 5.5 可扩展性
1. 连接器插件化：后续可扩展Kafka、PostgreSQL、API等数据源/目标端
2. 转换函数插件化：支持自定义Zig函数扩展计算能力
3. 后续可平滑升级为分布式集群模式

## 六、核心数据模型
### 6.1 系统元数据表（内置SQLite）
1. `datasource`：数据源表（id、mall_id、连接信息、创建时间、状态）
2. `sync_task`：同步任务表（id、数据源id、源表、目标表、转换规则、状态）
3. `sync_position`：同步位点表（任务id、GTID位点、binlog文件名、位点偏移、更新时间）
4. `alarm_rule`：告警规则表
5. `operation_log`：操作审计日志表
6. `reconcile_record`：对账记录表

### 6.2 目标归集库业务表
与业务对齐，核心三张宽表：
1. **`union_all_order` 全渠道订单表**
   - 核心字段：`id、mall_id、order_no、agent_id、order_total、agent_commission、commission_rate、order_status、pay_time、sync_time、is_delete`
   - 唯一索引：`UNIQUE KEY uk_mall_order(mall_id, order_no)`
2. **`union_all_user` 全渠道用户表**
   - 核心字段：`mall_id、user_id、agent_id、phone、register_time、sync_time`
3. **`error_order` 脏数据表**
   - 同订单表结构 + `error_msg、create_time`

## 十、风险与应对方案
1. **风险**：Zig生态缺少成熟的MySQL binlog原生库
   - 应对：优先基于成熟C库`mysql-binlog-connector-c`做Zig绑定，保证稳定性；核心协议逐步原生重写
2. **风险**：Zig 0.17版本语法/标准库迭代带来兼容性问题
   - 应对：锁定Zig 0.17正式版本编译，封装基础工具层隔离标准库变更
3. **风险**：百级任务并行下内存碎片化
   - 应对：实现内存池复用，批量分配与释放，避免频繁系统调用
4. **风险**：复杂表达式计算能力不足
   - 应对：V1.0先支持常用算术与条件表达式，高级规则通过自定义Zig函数扩展
